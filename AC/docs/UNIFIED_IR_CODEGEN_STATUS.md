## Known Issues ⚠️

### ASM/BNY Backend Limitations
The IR-based ASM generation has issues because:
1. **No stack offset tracking**: IR uses variable names/IDs but doesn't track stack frame offsets
2. **No register allocation**: IR doesn't assign variables to registers or memory locations
3. **Missing variable resolution**: Variables appear as `<?>` in generated ASM

**Root Cause**: The old ASM codegen tracked:
- `varOffsets`: map of variable name → stack offset (-8, -16, -24...)
- `stackOffset`: current stack frame position
- Allocated variables with `allocateStackVariable()` during code generation

**Solution**: This requires Phase 3+ work from the spec:
- Task 6: Control Flow Graph (CFG) lowering
- Task 7: Liveness analysis
- Task 8: Register allocation
- Task 9: x86-64 machine code generation

### Temporary Workaround
For now, ASM/BNY can fall back to the old AST-based codegen in `relics/` if needed.

## Implementation Details

### IR Opcodes Supported
- ✅ LABEL, JUMP, JUMP_IF_TRUE, JUMP_IF_FALSE
- ✅ STORE_VAR, LOAD_VAR (high-level backends only)
- ✅ ADD, SUB, MUL, DIV, MOD
- ✅ EQ, NEQ, LT, GT, LTE, GTE
- ✅ CALL, RETURN
- ✅ PRINT, LIB_CALL (Term.display)
- ✅ HALT, NOP
- ❌ LOAD_CONST (implicit in operands)
- ❌ FUNC_BEGIN, FUNC_END (handled by IRFunction structure)
- ❌ INPUT, ALLOC, FREE, LOAD_INDEX, STORE_INDEX (not yet generated by IR)
- ❌ EVENT_BIND, EVENT_TRIGGER (not yet generated by IR)

### Backend-Specific Code Generation
Each backend has custom handling for:
- **Headers/imports**: Language-specific includes
- **Function declarations**: `def`/`function`/`fn`/`func` syntax
- **Main entry point**: `if __name__`, `main()`, etc.
- **Type syntax**: `let`/`var`/`auto`/type inference
- **Print statements**: `print()`/`console.log()`/`println!()`/etc.

### Code Quality
- Clean if/else chains for backend selection
- No hardcoded backend specs (unlike ChatGPT's attempt)
- Consistent indentation handling
- Proper comment syntax per backend (`;` for ASM, `#` for Python, `//` for C-family)
